Tasks: Creation and Control

1. What Is a Task?

Section titled “1. What Is a Task?”

A task is a C function that never returns. It runs in an infinite loop, does its job, and then sleeps until it’s time to work again. Each task gets its own stack, its own priority, and the illusion of having the CPU all to itself.

Think About This

Why for (;;) and not while (1)? They’re identical — FreeRTOS examples use for (;;) by convention, but either works. What matters is that you never return from this function. If you do, FreeRTOS has no cleanup mechanism — the behavior is undefined and usually a crash.

3. Choosing a Stack Size

Section titled “3. Choosing a Stack Size”

Every task gets its own stack — it holds local variables, function call frames, and saved CPU registers during context switches. Get this wrong and you’ll either waste RAM or crash.

Words, Not Bytes

On ARM Cortex-M, usStackDepth is in words (4 bytes each). So 256 means 1024 bytes, and 512 means 2048 bytes. This catches everyone at least once.

Things That Eat Stack Fast

Section titled “Things That Eat Stack Fast”
  • Large local arrays: char buf[256] inside a task eats 64 words on every call. Move large buffers to static storage.
  • Recursion: Each recursive call adds a full stack frame. Avoid it in tasks.
  • printf / sprintf: These can use 200+ bytes of stack internally. Use snprintf with small buffers, or move the buffer to static storage.
  • Deep call chains: If your task calls A() which calls B() which calls C(), all their local variables are on the stack simultaneously.

Think About This

You create a task with 256 words of stack. Inside it, you declare char lcdLine[128]; — that’s 32 words gone just for one variable. Add the context switch overhead (~16 words on Cortex-M4), function call frames, and you’re already close to the edge. Always account for the biggest function in the call chain, not just the task function itself.

Section titled “4. Delays: vTaskDelay vs vTaskDelayUntil”

Both functions block the calling task (moving it to the Blocked state, using zero CPU). But they handle timing differently.

vTaskDelayUntil(&lastWake, ticks) — Absolute / Periodic

Section titled “vTaskDelayUntil(&lastWake, ticks) — Absolute / Periodic”

Sleeps until an absolute tick count, compensating for how long the work took:

Use when: the period must not drift (timekeeping, sampling, control loops).

What if the work takes longer than the period?

If doWork() takes 60 ms but your period is 50 ms, the next wake time is already in the past. The task won’t sleep at all — it’ll run again immediately, potentially starving lower-priority tasks. Always verify that your work completes well within the period.

5. Suspend and Resume

Section titled “5. Suspend and Resume”

Sometimes you need to pause a task completely — not just delay it. vTaskSuspend() moves a task to the Suspended state, where the scheduler ignores it entirely until you call vTaskResume().

A task can also suspend itself:

Suspend vs Delay

vTaskDelay is a timed block — the task wakes automatically. vTaskSuspend is indefinite — the task stays frozen until someone explicitly calls vTaskResume. Use suspend for manual control, delays for periodic timing.

6. Deleting Tasks

Section titled “6. Deleting Tasks”

When you delete a task, FreeRTOS frees its stack and TCB memory (the idle task handles the cleanup). Use this for tasks with a finite job — like a one-shot calibration or initialization sequence.

Don't Overuse Deletion

In most embedded systems, tasks run for the entire lifetime of the application. Creating and deleting tasks repeatedly fragments the FreeRTOS heap. If you need a task to “stop” temporarily, prefer vTaskSuspend over deletion.

Think About This

The Controller task has priority 2, Blink has priority 1. Why? Because when Controller calls vTaskSuspend(hBlink), it needs to preempt Blink immediately. If both had priority 1, the suspend would only take effect at the next tick — a subtle timing difference.

Caution

Do not call any blocking FreeRTOS API from inside an interrupt service routine. ISRs must use the ...FromISR variants exclusively. See the Queues guide and Semaphores guide for ISR-safe patterns.
